# BCM2 — 4H0907064 (Audi D4 / MLB onboard-supply control unit 2)

> Module-by-module Component-Protection (CP) documentation. The firmware code in the flashware is AES-encrypted, so this page documents **container structure, security-access (SA2), and memory layout** plus the **bench-dump plan** — it is not firmware reverse-engineering. Every privileged operation (read, flash, CP routine) is gated behind SA2; the actual CP/immo logic and per-vehicle state are recoverable only from an on-chip MCU dump.

BCM2 holds the CP/immobilizer/coding logic in P-Flash and the per-vehicle CP/immo/CS (challenge-secret)/coding state in D-Flash/EEPROM. Because the firmware is AES-locked in the flashware, both the CP logic and the live per-vehicle state require a bench read.

## 2. Flashware container (SGO / "SGML Object File")

The distributable is an **`.sgo` container** (e.g. `4H0907064___0582_S.sgo`). Each container is a sequence of addressed blocks tagged with a **crypt byte** that classifies the block:

| Crypt byte | Meaning |
|---|---|
| `0x00` | Plain header / flash-driver parameter table (XOR-0xFF only) — the only plaintext block |
| `0x01` | AES-encrypted code, **or** the bulk data-flash image |
| `0x0B` | AES-encrypted 128-byte page-erasable **data-flash** page (per-vehicle EEPROM) |

**For 4H0907064 (post-2013 AES body module) only the tiny container header at `0x000000` is extractable in plaintext** (576–960 B depending on SW; metadata/IDENT + the flash-routine parameter descriptor, **not** CPU code). Every genuine code block (`0x008000`–`0x0Fxxxx`) and every EEPROM/data block (`0xFFDE00`–`0xFFEFDF`) is AES-encrypted with no extractable key. The `.sgo` therefore yields only:

- the **block map** (addresses, sizes, crypt classes),
- the **SA2 security-access bytecode** (carried in container metadata),
- the **SW build IDENT** (e.g. `'0827'` at header offset `0xB1`).

**Contrast with BCM1 (4H0907063):** the sibling ships plain/keyless, so its firmware is directly readable from the flashware. BCM2 is not — bench MCU dump is mandatory.

**Family summary**

- **Entire family is AES-locked — no CP goldmine.** The only block that ever decodes to real plaintext is the container header at `0x000000`.
- **SA2 is constant across all 15 files**: byte-identical `6803814a05870f5c33554984663256cc4c`. One key routine unlocks the whole part number regardless of installed SW — solve it once.
- **Structure trend:** 14 of 15 files are the standard ~798 KB / 57-block layout (1 plain header + 56 AES: code `0x008000`–`0x0Bxxxx` in 8/16 KB chunks + the high EEPROM region). **0827_S** is the outlier: 1032 KB / 71 blocks, uniform 16 KB code sectors, code region extends to `0x0FC000` — a larger build, same 24-bit space, same AES scheme, same SA2.
- **`_E` variants** (0433_E, 0560_E) are structurally identical to their `_S` siblings (same block count, SA2, sizes).
- **Known false positive:** the unpacker flagged 0827_S block #9 @ `0x014000` as `bcb-xor` "csum-OK". This produced **zero bytes** — a checksum-oracle false positive (`sum(b'') & 0xFFFFFF == 0 == chk` with a garbage 26-byte key), **not plaintext**. It is AES like its idx-3 twin. Ignore it.

## 4. SA2 security access (UDS service `0x27`)

SA2 is the **SecurityAccess gate** for every privileged operation on BCM2: programming-session entry, `RequestUpload`/`RequestDownload` (block read or flash), erase, and **every `RoutineControl 0x31`** — including the Component-Protection adapt/clear/pair routines and CP-record reads/writes. Nothing proceeds until the ECU issues a seed (`0x27` sub `0x11`), you answer with the correct key (`0x27` sub `0x12`), and access is granted.

**Bytecode (17 bytes):**

```
6803814a05870f5c33554984663256cc4c
68 03 | 81 | 4A 05 | 87 0F5C3355 | 49 | 84 663256CC | 4C
```

Decoded with Simos-Suite's `Sa2SeedKey` interpreter (the bri3d / Brian Ledbetter opcode set that `uds_flash.py` drives via `_make_security_algo` → `Sa2SeedKey(sa2_script, seed_int).execute()`). The VW SA2 machine is a single 32-bit register + carry flag; the seed is loaded into the register and transformed into the key.

**Listing**

```
 0:  FOR   3              ; 0x68 0x03 -> loop body runs 3 times
 2:  RSL                  ; 0x81 rotate register left 1 bit through carry; carry := old bit31
 3:  BCC   +5  (->ip 10)  ; 0x4A 0x05 if carry==0 skip the EOR
 5:  EOR   0x0F5C3355     ; 0x87 register ^= 0x0F5C3355 (only when the rotated-out bit was 1)
10:  NEXT                 ; 0x49 loop back to ip 2 while iterations remain
11:  SUB   0x663256CC     ; 0x84 register -= 0x663256CC (mod 2^32)
16:  FINISH               ; 0x4C return register as the 4-byte key
```

**Algorithm:** load seed → for i in 0..2: rotate-left-through-carry by 1; if the bit shifted out of bit31 was 1, XOR with `0x0F5C3355`; end loop; finally subtract `0x663256CC` (mod 2³²). Return the register. This is an LFSR/CRC-style bit mixer followed by a constant subtract — **pure arithmetic, no per-vehicle/HW secret, no AES.** The two constants `0x0F5C3355` / `0x663256CC` **are** the shared secret.

**Verified seed → key pairs (real interpreter):**

| seed | key |
|---|---|
| `0x00000000` | `0x99CDA934` |
| `0x11223344` | `0x22DF4354` |
| `0xDEADBEEF` | `0x707305B4` |
| `0xFFFFFFFF` | `0x6D391088` |
| `0x01020304` | `0xA1DDC154` |
| `0x80000000` | `0xD73E7684` (exercises the carry/EOR path) |
| `0x40000000` | `0xB8860FDC` |

**On the bench:** request seed via UDS `SecurityAccess 0x27` (programming-session level `0x11` per `client.unlock_security_access(0x11)`; seed sub-func `0x11`, send-key sub-func `0x12`), run this script over the 4-byte seed, reply with the 4-byte key.

**Scope:**
- **Does not vary by SW** — all 15 files carry the identical script. Exactly 1 distinct SA2 across the whole part number.
- **Per-module / per-component-family, not universal** — the constants are specific to 4H0907064. BCM1 (4H0907063) carries a different SA2 (`68119391570412871a4743f6494c`) and ships non-AES (plain) flashware; the J533 gateway, Simos ECUs and HVAC each carry their own distinct SA2 with different constants.
- **Fully satisfiable offline** — fixed-constant, secret-free, SW-independent. No online/GeKo seed-key server is needed; the part on the bench (or over OBD) suffices.
- **SA2 is the access gate only — it does NOT decrypt the AES code blocks** in the `.sgo`. That key is separate, which is why this remains structure documentation pending an MCU dump.

## 5. Memory map + CP/immo data region

24-bit address space. Source: `cp_tools.sgo_unpack` on `4H0907064___0827_S.sgo` (71 blocks, fullest) cross-checked against `0582_S` (57 blocks) and confirmed against all 13 `_S` variants.

| Region | Address range | Kind | Crypt | Contents |
|---|---|---|---|---|
| **Plain header / flash-driver param table** | `0x000000`–`0x00024F` (592 B on 0827 / 576 B on 0582) | header | `0x00` | The only plaintext block. u16-LE record table with `0x80`-prefixed segment markers; embeds the SW build string (`'0827'` @ `0xB1`). Flash-routine/parameter descriptor + IDENT — **not** vehicle data, **not** resident CPU code. Factory-uniform across every `_S`. SA2 bytecode is carried in container metadata here. |
| **P-Flash application code (low window)** | `0x008000`–`0x0FFFFF` (0827_S, full 1 MB) / `0x008000`–`0x0BFFFF` (0096–0582, 768 KB) | pflash-code | `0x01` | AES-encrypted program flash. 0827_S: uniform 16 KB sectors, 64 code blocks. 0582_S: mixed 8/16 KB sectors, 47 code blocks, code-top `0x0C0000`. Holds the V850 application incl. the **CP/immo/coding logic** and the **SA2 security state machine**. 0827_S contains a genuine **duplicate write** of `0x010000`–`0x017FFF` (idx 2/3 then 8/9) — a repeated SWDL pass (CBOOT/shadow or code+verify two-pass); the idx-9 `bcb-xor` label is the AES false positive. Structure only, no plaintext. |
| **D-Flash / EEPROM pages (per-vehicle CP/immo/CS/coding)** | `0xFFDE00`–`0xFFDFFF` (5 pages) + `0xFFE000`–`0xFFE07F` (1 page) = six 128-B pages | dflash-eeprom | `0x0B` | Page-organized data-flash EEPROM. Each erase/prog range = exactly one 128 B page (`0xFFDE00`/`E80`/`DF00`/`DF80`/`E000`). **Where per-vehicle CP / immobilizer / CS (challenge-secret) / coding-adaptation state lives at runtime.** Identical footprint in all 13 `_S` variants. The `.sgo` ships only factory-default, AES-encrypted content — live per-car values exist only on the installed chip. |
| **D-Flash / EEPROM bulk image** | `0xFFE080`–`0xFFEFDF` (`0xF60` = 3936 B) | dflash-eeprom | `0x01` | Largest high-region record (~3.9 KB), contiguous above the 128-B page set. Image-style (not `0x0B` page-style) — a single bulk data-flash image. Holds the bulk coding/adaptation/long-coding + extended CP/immo data set. Identical address/size across variants; AES factory default in the `.sgo`. |
| **Boot / vector region** | `0x000000`–`0x007FFF` | boot | — | **Not present as an `.sgo` block.** On the V850-class part the reset vector + bootloader/CBOOT live here and are **not** field-flashable via the `.sgo` (factory/ISP-locked). Application code starts at `0x008000`, consistent with a ~32 KB protected boot/CBOOT area. Recoverable only by a full-chip bench read. |

**Unmapped gap:** `0x100000`–`0xFFDDFF` is not flashed.

**CP/immo region (summary):** the high data-flash/EEPROM at **`0xFFDE00`–`0xFFEFDF`** (top of the 24-bit space), in two sub-parts:
- **(a)** six 128-B page-erasable records `0xFFDE00`–`0xFFDFFF` + `0xFFE000`–`0xFFE07F`, crypt `0x0B` — the **per-vehicle CP/immo/CS state pages**;
- **(b)** one `0xF60`-byte bulk image `0xFFE080`–`0xFFEFDF`, crypt `0x01` — the **coding/adaptation store**.

The `.sgo` only ships AES-encrypted factory defaults for this whole region, so the live per-vehicle CP/immo/CS/coding values exist **solely on the installed chip** and must be read off the part on the bench.

## 6. Bench-dump target list

What to read once bench MCU access is set up, and why. Keep P-Flash and D-Flash as **separate images** — same chip, distinct partitions and read addresses. **SA2 (Section 4) is the precondition for every read below.**

**Target 1 — Full P-Flash read `0x000000`–`0x0FFFFF` (1 MB), single contiguous read.**
Captures (a) the protected boot/CBOOT + reset vectors at `0x000000`–`0x007FFF` that the `.sgo` never ships, and (b) the AES-decrypted, **as-running application code** `0x008000`–`0x0FFFFF` — the CP/immo/coding logic and the SA2 state machine. *Why bench:* code blocks are AES (key-less) in the `.sgo`, so on-chip is the only route to plaintext firmware for RE. On a 0096–0582 unit the populated code reaches only `0x0C0000`, but **read the full 1 MB anyway** to catch the boot area and any tail data.

**Target 2 — D-Flash / EEPROM read `0xFFDE00`–`0xFFDFFF` + `0xFFE000`–`0xFFE07F` (six 128-B pages, crypt `0x0B`). PRIMARY CP/immo target.**
These pages hold the per-vehicle **Component Protection / immobilizer / CS challenge-secret** state. The `.sgo` ships only encrypted factory defaults; live values are on-chip only. Read as **data-flash, not P-flash** (separate partition / different read command on the V850).

**Target 3 — D-Flash / EEPROM read `0xFFE080`–`0xFFEFDF` (`0xF60` = 3936 B, crypt `0x01`).**
The bulk coding/adaptation + extended CP/immo store. Read together with the `0x0B` pages — they form one contiguous high-region EEPROM image `0xFFDE00`–`0xFFEFDF`.

**Practical framing (Xhorse/VVDI-style two-pass):**
1. **P-Flash read** → dump `0x000000`–`0x0FFFFF` as one `.bin` (the V850 R5F36xx flash array).
2. **D-Flash/EEPROM read** → dump the high region `0xFFDE00`–`0xFFEFE0` (~4.5 KB) as a separate `.bin` — this is the small per-vehicle blob to compare car-to-car.

Diff the D-flash dump against the `.sgo` factory default (once a key exists) or against a virgin/donor unit to isolate the CP/immo/CS bytes.

**Reference baseline:** for each dump, keep the matching `.sgo` (e.g. `4H0907064___0827_S.sgo`) so the on-chip read can be diffed against the factory-default region once decrypted. The SA2 (`6803814a05870f5c33554984663256cc4c`) and the code-window size (`0x100000` for 0827 vs `0x0C0000` for 0096–0582) identify which SW the dumped unit is running.

## 7. Status (honest)

- The **firmware code is AES-locked** in every BCM2 `.sgo`. Across all 15 SW files, the only block that decodes to real plaintext is the container header at `0x000000` (576–960 B of metadata/IDENT, not MCU code). Every genuine code block and every EEPROM/data block is AES-encrypted with **no extractable key**.
- The `.sgo` route therefore yields **structure documentation only**: the block/memory map, the SA2 security-access bytecode (Section 4), and the SW layout. This is exactly what is expected for post-2013 AES body modules.
- The lone apparent exception (0827_S block #9 `bcb-xor`) is a **zero-byte checksum-oracle false positive**, not plaintext.
- **CP reverse-engineering requires the bench MCU dump** (Section 6): the `.sgo` cannot be turned into plaintext firmware for any SW in this family. The P-Flash dump yields the CP/immo logic + SA2 state machine; the D-Flash dump yields the per-vehicle CP/immo/CS/coding state.
- **SA2 is solved and SW-independent** — the gate that fronts every read/flash/CP routine is fully characterized and satisfiable offline with the part on the bench (or over OBD). It is the access gate only; it does **not** decrypt the AES code blocks (separate key), which is why firmware RE still waits on the dump.
